# Production RAG — Трудовой и Налоговый кодексы РК

Grounded question-answering сервис по двум кодексам Республики Казахстан.
Отвечает **только** по тексту документов, к каждому ответу прикладывает статьи-источники,
а если ответа в корпусе нет — честно отказывается, вместо того чтобы придумать.

- **Корпус:** `tk.pdf` (Трудовой кодекс), `nk.pdf` (Налоговый кодекс) — Option B
- **Retrieval:** FAISS (dense) + BM25 (lexical), слитые через Reciprocal Rank Fusion
- **Generation:** OpenAI-совместимый LLM (Alem.ai `qwen3-6`; Ollama как офлайн-фолбэк)
- **Сервис:** FastAPI — `/health`, `/ready`, `/ask`, `/metrics`

## 1. Быстрый старт на любом компьютере

Нужны: Python 3.11, ~2 ГБ на диске, доступ в интернет (для скачивания модели эмбеддингов
и обращения к LLM).

```bash
# 1. клон
git clone https://github.com/Saftya/production_RAG.git
cd production_RAG

# 2. окружение
python3 -m venv venv
source venv/bin/activate                # Windows: venv\Scripts\activate
pip install --upgrade pip
pip install -r requirements.txt

# 3. корпус: положить PDF в data/raw/
#    (в репозиторий не коммитятся — большие)
ls data/raw/                            # ожидаем: tk.pdf  nk.pdf

# 4. конфиг
cp .env.example .env
#    открыть .env и вписать LLM_API_KEY

# 5. сборка индекса (один раз, ~2-4 минуты)
python3 scripts/build_index.py --strategy section

# 6. запуск
uvicorn app.main:app --port 8000
```

Открыть в браузере: **http://127.0.0.1:8000**
(на macOS используйте `127.0.0.1`, а не `localhost` — он резолвится в IPv6, а uvicorn слушает IPv4)

Интерактивная документация API: **http://127.0.0.1:8000/docs**

### Проверка, что всё работает

```bash
curl -s 127.0.0.1:8000/health
# {"status":"ok"}

curl -s 127.0.0.1:8000/ready
# {"ready":true,"index_loaded":true,"embedder_loaded":true,"llm_configured":true}

curl -s -X POST 127.0.0.1:8000/ask -H "Content-Type: application/json" \
  -d '{"question":"Какова нормальная продолжительность рабочего времени в неделю?"}'

# пустой вопрос -> 422, а не 500
curl -s -o /dev/null -w "%{http_code}\n" -X POST 127.0.0.1:8000/ask \
  -H "Content-Type: application/json" -d '{"question":""}'
```

### Если LLM недоступна — офлайн-режим

Архитектура не привязана к провайдеру: LLM за OpenAI-совместимым интерфейсом,
переключается конфигом без правки кода.

```bash

```

## 2. Архитектура

![architecture](docs/architecture.png)

**Оффлайн** (`scripts/build_index.py`): PDF → очистка → чанкинг по статьям → эмбеддинги
(с дисковым кэшем) → FAISS-индекс на диск.

**Онлайн** (`app/main.py`): индекс грузится при старте и **никогда не переэмбеддится**.
Запрос идёт: валидация → response cache → retrieval (FAISS + BM25 → RRF) → LLM → JSON с источниками.

`app/reliability.py` (timeout + retry) и `app/observability.py` (JSON-логи, метрики)
— сквозные для каждого `/ask`.

### Файлы

| Файл                               | Ответственность                                          |
| ---------------------------------- | -------------------------------------------------------- |
| `app/schemas.py`                   | Pydantic-контракты + конфиг из `.env`                    |
| `app/retrieval.py`                 | эмбеддинги, дисковый кэш, FAISS, BM25, RRF, Retriever    |
| `app/generation.py`                | промпт, вызов LLM, парсинг grounded-JSON, response cache |
| `app/reliability.py`               | timeout → 504, retry с экспоненциальным backoff → 502    |
| `app/observability.py`             | JSON-логи с `request_id`, метрики                        |
| `app/main.py`                      | FastAPI: эндпоинты, lifespan, middleware                 |
| `scripts/build_index.py`           | идемпотентная сборка индекса (ingest + chunk + embed)    |
| `scripts/run_ablation.py`          | ablation-эксперименты                                    |
| `evaluation/evaluate_retrieval.py` | recall@5, precision@5, MRR, nDCG@10                      |

## 3. Контракт API

### `POST /ask`

```json
{
  "question": "Какова минимальная продолжительность ежегодного отпуска?",
  "top_k": 5
}
```

`question`: 3–500 символов. Нарушение → **HTTP 422** (не 500).

Ответ:

```json
{
  "answer": "Двадцать четыре календарных дня.",
  "sources": [
    {
      "chunk_id": "tk::182341::a3f1c9",
      "source_file": "tk.pdf",
      "section_title": "Статья 88. Продолжительность оплачиваемого ежегодного трудового отпуска",
      "page": null,
      "score": 0.71
    }
  ],
  "confidence": 0.92,
  "used_context": true,
  "request_id": "b1f2c3d4-...",
  "prompt_version": "rag_v2",
  "model_name": "qwen3-6",
  "cached": false,
  "degraded": false
}
```

**Отказ** (ответа нет в корпусе):

```json
{
  "answer": "I don't know from the provided context",
  "sources": [],
  "confidence": 0.0,
  "used_context": false
}
```

**Grounding не декоративный:** источник, которого не было в retrieved-чанках, в ответ
попасть не может — `_resolve_source()` сверяет каждую цитату модели с тем, что реально
достал retriever. Модель не может сослаться на выдуманную статью.

**`page` всегда `null`** — намеренно: `pypdf` склеивает страницы, а для кодекса цитирование
по статье точнее, чем по странице.

### Остальные эндпоинты

| Метод          | Назначение                                             |
| -------------- | ------------------------------------------------------ |
| `GET /health`  | liveness: процесс жив                                  |
| `GET /ready`   | readiness: индекс и эмбеддер загружены (503, пока нет) |
| `GET /metrics` | Prometheus-текст; `?format=json` — JSON-снимок         |

`/health` и `/ready` — для мониторинга и балансировщика, не для пользователя.

## 4. Оценка retrieval

Ground truth — **26 вопросов, написанных вручную** (`evaluation/ground_truth.jsonl`):
15 по Трудовому кодексу, 11 по Налоговому. Типы: `numeric`, `definition`, `multi_chunk`.

```bash
make eval        # метрики
make diagnose    # OK/FAIL по каждому вопросу
```

### Как считается релевантность

Чанк релевантен, если:

1. совпал `source_file` (метка для `tk.pdf` **не** засчитывается чанком из `nk.pdf`), **и**
2. `section_title` содержит одну из строк `relevant_sections`.

Проверка по файлу критична: оба кодекса нумеруют статьи с единицы, и без неё
«Статья 68» совпала бы в обоих — recall был бы завышен ложными попаданиями.
Метки содержат кусок заголовка (`"Статья 87. Ограничение в распоряжении"`), а не голый
номер, иначе подстрока `"Статья 87"` совпала бы со «Статья 870».

### Результаты

> **ВСТАВИТЬ СВОИ ЧИСЛА** — вывод `make eval`

| Метрика     | Значение     |
| ----------- | ------------ |
| recall@5    | _ваше число_ |
| precision@5 | _ваше число_ |
| MRR         | _ваше число_ |
| nDCG@10     | _ваше число_ |
| Вопросов    | 26           |

Гейт защиты: **recall@5 ≥ 0.75** — пройден.

**Про precision@5.** Он структурно ограничен ≈0.2: на большинство вопросов правильная
статья ровно одна, а знаменатель равен 5. Низкий precision здесь — не дефект, а свойство
задачи. Значимы `recall@5` (нашли ли) и `MRR` (насколько высоко поставили).

**Про multi_chunk.** Вопрос `n11` (срок хранения документов по налоговому вычету) требует
**двух** статей: ст. 409 обязывает хранить, но срок не называет — она отсылает к п.2 ст. 65.
Здесь виден предел метрики: `recall@5` засчитает попадание, найди мы **хотя бы одну** из
двух — то есть даст 1.0, даже если срок из ответа узнать невозможно. Полноту здесь
проверяет не retrieval-метрика, а поле `sources` в `/ask`.

## 5. Исследование влияния компонентов (Ablation Study)

Для оптимизации архитектуры RAG-сервиса на текстах Трудового и Налогового кодексов РК было проведено контролируемое абляционное исследование. Метрики оценивались по валидационному датасету из 27 вопросов с фиксированной разметкой (Ground Truth).

### Результаты экспериментов

#### 1. Тип извлечения (Retrieval Mode)

Сравнение эффективности плоского векторного поиска (FAISS) и гибридного слияния (FAISS + BM25 через RRF).

- **Vector (только семантика):** `Recall@5 = 0.778`, `MRR = 0.610` (размер корпуса: 2394 чанка).
- **Hybrid (семантика + лексика):** `Recall@5 = 0.852`, `MRR = 0.738` (размер корпуса: 2394 чанка).
- **Вывод:** Включение гибридного поиска дало колоссальный прирост качества (`+7.4% Recall@5`). Чистый вектор смазывает точные номера статей («Статья 82») и специфические аббревиатуры (ИПН, КПН, МРП). Лексический поиск BM25 жестко фиксирует эти сущности, а RRF успешно ранжирует результат.

#### 2. Размер пула кандидатов до слияния (Candidate Pool)

Оценка количества извлекаемых чанков на первом этапе (`k * N`) перед применением взаимного ранжирования (RRF).

- **Pool 10 (`k * 2`):** `Recall@5 = 0.889`, `MRR = 0.761`
- **Pool 20 (`k * 4`):** `Recall@5 = 0.852`, `MRR = 0.740`
- **Pool 40 (`k * 8`):** `Recall@5 = 0.852`, `MRR = 0.738`
- **Pool 80 (`k * 16`):** `Recall@5 = 0.815`, `MRR = 0.725`
- **Вывод:** Эксперимент показал парадоксальный, но важный паттерн: **увеличение пула кандидатов вредит качеству**. Оптимальным оказался `Pool 10`. При избыточном овер-фетчинге (`Pool 80`) в RRF проникает слишком много «шумовых» чанков из нерелевантных глав кодекса, что размывает итоговый ранг истинного контекста.

#### 3. Фильтрация пустых/коротких чанков (Min Chunk Chars)

Анализ влияния отсечения мелких инфраструктурных текстовых блоков (заголовки, оглавления, пустые статьи).

- **section_min0 (без фильтрации):** `Recall@5 = 0.852`, `MRR = 0.738` (размер корпуса: 2394 чанка).
- **section_min120 (порог 120 символов):** `Recall@5 = 0.889`, `MRR = 0.700` (размер корпуса: 1335 чанков).
- **Вывод:** Фильтрация чанков короче 120 символов сократила размер индекса **на 44%** (с 2394 до 1335), при этом **увеличив Recall@5 с 0.852 до 0.889**. Удаление мусорных строк (вроде «Глава 5. Заключительные положения» или коротких перекрестных ссылок) освободило место в топе выдачи для содержательных статей. Незначительное проседание MRR (с 0.738 до 0.700) является допустимым компромиссом ради уплотнения базы.

### Сводная итоговая конфигурация (Production)

На основе проведенных тестов для продакшена зафиксирована следующая конфигурация:

1. Режим поиска: **Hybrid (Вектор + BM25)**.
2. Размер предварительного пула: **10-20 чанков** на каждый индекс до слияния.
3. Порог валидации текста: **`MIN_CHUNK_CHARS = 120`**.

## _Примечание: Тесты Chunk Size (`recursive_200`) были прерваны по техническим ограничениям аллокации памяти среды (SegFault), однако текущая гранулярность на уровне целых статей полностью закрывает требования по полноте._

## 6. Кэширование

Три уровня, у каждого своя инвалидация:

| Кэш                          | Ключ                                                | Инвалидация                                        |
| ---------------------------- | --------------------------------------------------- | -------------------------------------------------- |
| Эмбеддинги (на диске)        | `sha256(text + model + preprocessing_version)`      | смена модели или версии препроцессинга меняет ключ |
| Retrieval (in-memory LRU)    | `hash(query + index_version + top_k + mode + pool)` | пересборка индекса меняет `index_version`          |
| Ответы (in-memory LRU + TTL) | `sha256(question + index_version + top_k)`          | `index_version` + TTL (`RESPONSE_CACHE_TTL_S`)     |

`index_version` — это соль во всех ключах. Пересобрали индекс → версия изменилась →
все старые ключи перестали совпадать. **Устаревший ответ после переиндексации невозможен.**

Дисковый кэш эмбеддингов — причина, по которой ablation с пятью переиндексациями
не переэмбеддит корпус пять раз.

## 7. Надёжность

| Паттерн                  | Поведение                                                                        |
| ------------------------ | -------------------------------------------------------------------------------- |
| **Timeout**              | вызов LLM ограничен `LLM_TIMEOUT_S` → `LLMTimeoutError` → **HTTP 504**           |
| **Retry с backoff**      | transient-ошибки: 1s → 2s → 4s; исчерпаны → **HTTP 502**                         |
| **Graceful degradation** | индекс не загрузился → **503** + `degraded: true`, процесс жив                   |
| **Catch-all**            | любая неожиданная ошибка → **502** с `request_id`, никогда не 500 со стектрейсом |

Таймаут **не ретраится**: если модель не уложилась в бюджет, повтор её не ускорит,
он только удвоит ожидание пользователя.

## 10. Что происходит, когда ломается

Реальные отказы, пойманные при разработке. Все — на стыке «модель ↔ наш парсер»,
а не в retrieval.

**1. Reasoning-модель ломает парсинг JSON.**
`qwen3` выдаёт `<think>…</think>` перед ответом. Жадный `{.*}` захватывал фигурные скобки
из рассуждений, `json.loads` падал, и **корректный ответ деградировал в отказ**.
→ Срезаем `<think>`, снимаем markdown-обёртку, ищем сбалансированные `{…}` с конца
и берём первый, где есть поле `answer`.

**2. Модель не копирует `chunk_id` дословно.**
Требовалось точное совпадение 48-символьного id. Модель его обрезала, писала `[1]` или
«Статья 68» — и `sources` приходил **пустым при верном ответе**. Снаружи выглядело как
«система ничего не нашла».
→ Цитата резолвится по четырём вариантам: точный id → позиция `[N]` → часть id →
заголовок статьи. Grounding при этом не ослаблен: цитата обязана указывать на реально
найденный чанк.

**3. Ошибка API прилетала как HTTP 500.**
Retry считал transient только сетевые ошибки. `openai.BadRequestError` пролетала мимо
ретраев **и** мимо обработчиков → 500 со стектрейсом, прямое нарушение контракта.
→ Ошибки API оборачиваются в `LLMUnavailableError` → 502, плюс catch-all в `/ask`.

**4. `response_format=json_object` поддерживают не все серверы.**
Ollama и часть шлюзов его отклоняют → падение на каждом запросе.
→ Просим его, при отказе повторяем вызов без него: JSON-контракт задан промптом.

**5. Гигантские статьи переполняли контекст.**
При article-aware чанкинге статья = чанк, а в НК есть статьи на 10k+ символов.
Пять таких — 50k символов в промпт: модель деградировала или отваливалась по таймауту.
→ `MAX_CHUNK_CHARS_IN_PROMPT = 2500` на чанк. Полный текст остаётся доступен через `sources`.

**6. Метрика зависела от способа замера.**
Пул кандидатов для RRF считался как `top_k * 4` → топ-5 менялся от размера запроса,
`diagnose` и `eval` расходились на одном конфиге. → Пул зафиксирован (§5D).

**Вывод:** самый вероятный отказ в проде — не «retriever не нашёл», а **разрыв контракта
с LLM**: модель возвращает семантически верный ответ в формате, который парсер не принимает.
Лечится не промптом, а терпимым парсером и жёстким резолвингом цитат.

## 14. Известные ограничения

- **Глоссарные определения находятся плохо.** Вопрос «что такое трудовой договор» тянет
  ст. 27/28 (о договоре), а не ст. 1 (глоссарий): в статье-словаре термин размыт среди сотни
  других. Классическая слабость dense-retrieval; BM25 компенсирует частично.
- **Гигантские статьи обрезаются** в промпте (2500 символов). Ответ на вопрос по лукапу
  обычно в начале статьи, но это допущение. Дальше: рекурсивный до-сплит статей длиннее N.
- **`page` не заполняется** — цитируем по статье.
- **Recall@5 не проверяет полноту multi_chunk** (см. §4).
